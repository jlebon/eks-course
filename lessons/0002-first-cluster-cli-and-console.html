Lesson 2: Your First EKS Cluster

~30 min hands-on · Requires AWS account · Creates real resources (cost ~$0.10/hr for control plane + node costs)

In this lesson you'll create a real EKS cluster, see it in the console, connect kubectl, verify it's working, and tear it down. You'll do this via eksctl first (fast path), then walk through the console to understand what was created for you.

💰 Cost warning An EKS cluster costs $0.10/hour for the control plane alone, plus EC2 costs for nodes (e.g. ~$0.04/hr for a t3.medium). Delete everything at the end of this lesson. Total cost for ~1 hour: approximately $0.20–$0.30.

Part 1: Tool Setup

You need three CLI tools. Let's verify (or install) each one.

1

AWS CLI

You should already have this configured with your work credentials.

# Check version (need 2.x)
aws --version

# Verify you're authenticated and see your identity
aws sts get-caller-identity

You should see your account ID, user ARN, etc. If not, run aws configure or set up your credentials file.

Source: AWS Docs: Install AWS CLI

2

kubectl

You likely have this already. Verify it's a recent version:

kubectl version --client

Any recent version works — kubectl supports ±1 minor version skew with the cluster.

3

eksctl

This may be new to you. Install it:

# Linux (amd64)
ARCH=amd64
PLATFORM=$(uname -s)_$ARCH
curl -sLO "https://github.com/eksctl-io/eksctl/releases/latest/download/eksctl_$PLATFORM.tar.gz"
tar -xzf eksctl_$PLATFORM.tar.gz -C /tmp && rm eksctl_$PLATFORM.tar.gz
sudo install -m 0755 /tmp/eksctl /usr/local/bin && rm /tmp/eksctl

# Verify
eksctl version

As of June 2026, the latest is v0.226.0. You need at least v0.215.0.

Source: AWS Docs: Install eksctl

Part 2: Create the Cluster (eksctl)

We'll use eksctl with a YAML config file. This is preferable to inline flags because the config is reproducible and self-documenting.

4

Write the cluster config

Create a file called cluster.yaml:

apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig

metadata:
  name: learn-eks          # pick any name
  region: us-east-2        # pick your preferred region
  version: "1.36"          # latest EKS version as of June 2026

managedNodeGroups:
  - name: default
    instanceType: t3.medium
    desiredCapacity: 2
    minSize: 1
    maxSize: 3

That's it. eksctl will create everything else for you: VPC, subnets, internet gateway, NAT gateway, route tables, cluster IAM role, node IAM role, security groups, the EKS cluster, and the managed node group.

ℹ️ Why managedNodeGroups and not nodeGroups? In eksctl config, managedNodeGroups creates AWS-managed node groups (AWS handles the ASG and rolling updates). nodeGroups creates self-managed node groups. For learning, managed is simpler. We'll switch to self-managed when we get to custom AMIs.
5

Dry-run first

See what eksctl would create without creating it:

eksctl create cluster -f cluster.yaml --dry-run

This outputs the full expanded config — all the defaults eksctl fills in. Skim it to see VPC CIDR, subnet layout, IAM policies, etc.

6

Create the cluster

eksctl create cluster -f cluster.yaml

This takes 15–20 minutes. While you wait, read Part 3 below — we'll explore what's happening in the console.

You'll see output like:

2026-06-09 10:00:00 [ℹ]  eksctl version 0.226.0
2026-06-09 10:00:00 [ℹ]  using region us-east-2
2026-06-09 10:00:02 [ℹ]  setting availability zones to [us-east-2a us-east-2b us-east-2c]
2026-06-09 10:00:02 [ℹ]  subnets for us-east-2a - public:192.168.0.0/19 private:192.168.96.0/19
...
2026-06-09 10:15:00 [✔]  EKS cluster "learn-eks" in "us-east-2" region is ready

When done, eksctl automatically updates your ~/.kube/config so kubectl points at the new cluster.

Part 3: Explore in the Console (while eksctl runs)

Open the EKS console while the cluster is being created. This teaches you what eksctl is doing behind the scenes.

7

Watch CloudFormation

Go to CloudFormation in the AWS console. You'll see two stacks being created:

Stack NameCreates
eksctl-learn-eks-clusterVPC, subnets, internet gateway, NAT gateways, route tables, cluster IAM role, the EKS cluster itself, security groups
eksctl-learn-eks-nodegroup-defaultNode IAM role, launch template, managed node group (Auto Scaling Group)

Click into each stack → Resources tab to see every individual AWS resource. This is what you'd have to create manually without eksctl.

Source: AWS Docs: Getting started with eksctl

8

Explore the EKS console

Once the cluster status shows Active (this happens before nodes are ready):

  1. Go to EKS → Clusters → learn-eks
  2. Overview tab — cluster endpoint URL, K8s version, platform version, cluster IAM role
  3. Compute tab — your managed node group (default), click into it to see instances
  4. Networking tab — VPC, subnets, security groups, cluster endpoint access (public/private)
  5. Add-ons tab — CoreDNS, kube-proxy, VPC CNI are auto-installed
  6. Resources tab — live view of Kubernetes objects (pods, services, etc.) right from the console

Spend a few minutes clicking around. The console is how you'll verify things when debugging node issues later.

Part 4: Connect and Verify

9

Verify kubectl connectivity

Once eksctl create cluster finishes:

# See your nodes
kubectl get nodes -o wide

# Expected output: 2 nodes in Ready state
NAME                                       STATUS   ROLES    AGE   VERSION
ip-192-168-xx-xx.us-east-2.compute.internal  Ready    <none>  2m    v1.36.x
ip-192-168-yy-yy.us-east-2.compute.internal  Ready    <none>  2m    v1.36.x
# See system pods
kubectl get pods -A

# You should see pods in kube-system namespace:
# - aws-node (VPC CNI) on each node
# - coredns (2 replicas)
# - kube-proxy on each node
10

Quick smoke test

Deploy nginx to prove the cluster works:

# Create a deployment
kubectl create deployment hello --image=nginx --replicas=1

# Check it's running
kubectl get pods -l app=hello

# Port-forward to access it locally
kubectl port-forward deployment/hello 8080:80

# In another terminal (or Ctrl-C after testing):
curl http://localhost:8080
# Should return the nginx welcome page HTML

# Clean up the test deployment
kubectl delete deployment hello

Part 5: What eksctl Created (the Manual Way)

If you'd done this via the console or aws eks CLI instead of eksctl, here's what you'd have to create yourself, in order:

#StepConsole location
1Create a VPC with public + private subnets in ≥2 AZsVPC → Create VPC
2Create an internet gateway, NAT gateway, route tablesVPC → Internet/NAT Gateways
3Create the cluster IAM role with AmazonEKSClusterPolicyIAM → Roles → Create role → EKS service
4Create the EKS cluster, selecting VPC, subnets, roleEKS → Create cluster
5Wait ~10 min for cluster to become Active—
6Update kubeconfig: aws eks update-kubeconfig --name learn-eks --region us-east-2CloudShell or local terminal
7Create the node IAM role with Worker, CNI, and ECR policiesIAM → Roles → Create role → EC2 service
8Create a node group, selecting instance type, scaling, subnets, roleEKS → Cluster → Compute → Add node group
9Wait ~5 min for nodes to join—

That's 9 manual steps across 3 different AWS services. eksctl does all of it in one command.

🔑 Key insight eksctl translates your YAML config into CloudFormation templates. If you ever need to understand or debug what was created, look at the CloudFormation stacks — they're the source of truth, not eksctl itself.

Source: AWS Docs: Getting started — Console and CLI

Part 6: The Console Create Flow (Guided Tour)

You don't need to do this now — but here's the console cluster creation flow so you know it when you need it. The EKS console is at console.aws.amazon.com/eks.

Console steps (click-by-click)

  1. Clusters → Add cluster → Create
  2. Configuration options → select Custom configuration, toggle off Use EKS Auto Mode
  3. Configure cluster:
    • Name — your cluster name
    • Kubernetes version — pick 1.36 (latest)
    • Cluster service role — select or create the cluster IAM role
    • Cluster access — Bootstrap admin access: Allow. Auth mode: EKS API (recommended)
  4. Specify networking:
    • VPC — pick your VPC (must have subnets in ≥2 AZs)
    • Subnets — select private subnets for nodes
    • Cluster endpoint access — "Public and private" for learning
  5. Configure observability → defaults are fine
  6. Select add-ons → defaults install CoreDNS, kube-proxy, VPC CNI
  7. Configure add-ons settings → defaults are fine
  8. Review and create → Create

Then, once the cluster is Active, go to Compute → Add node group and walk through the node group wizard (node role, instance type, scaling, subnets).

ℹ️ When to use the console vs eksctl Use eksctl when you want reproducibility and speed. Use the console when you want to inspect what exists, debug issues, or when your org requires console-based workflows. Both create the same underlying resources.

Part 7: Clean Up

💰 Do this now! Don't leave the cluster running. Delete it to avoid charges.
11

Delete everything

eksctl delete cluster -f cluster.yaml

# Or equivalently:
# eksctl delete cluster --name learn-eks --region us-east-2

This deletes the CloudFormation stacks, which tears down nodes, node group, cluster, IAM roles, VPC, subnets, gateways — everything.

Takes ~5–10 minutes. Verify in CloudFormation that both stacks show DELETE_COMPLETE.

12

Verify cleanup in console

Check these console pages to confirm nothing is left behind:

  • EKS → Clusters — should be empty
  • CloudFormation → Stacks — both stacks should be gone (or show DELETE_COMPLETE)
  • EC2 → Instances — no leftover instances from the node group
  • VPC → Your VPCs — the eksctl-created VPC should be deleted
