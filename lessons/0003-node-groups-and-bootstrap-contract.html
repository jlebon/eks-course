Lesson 3: Node Groups & the Bootstrap Contract

~15 min · No AWS account needed · Directly relevant to your bootc AMI goal

This lesson answers the question at the heart of your mission: what exactly does an AMI need to do to become an EKS node? We'll cover the two node group types in detail, then dissect the bootstrap contract — the specific requirements your bootc AMI must satisfy.

Managed vs Self-Managed: The Full Picture

You know the four compute options from Lesson 1. For custom AMI work, only two matter: managed node groups and self-managed node groups. Here's the complete comparison:

Aspect Managed Node Group Self-Managed Node Group
Who creates the ASG? AWS (via EKS API) You (via CloudFormation, eksctl, or Terraform)
Rolling updates? AWS handles — respects PDBs, drains nodes You handle — manual or via tooling
Custom AMI? Yes, but must declare an amiFamily from: AL2023, AL2, Bottlerocket, Ubuntu Yes, any AMI — no family restriction
User data handling AWS merges your user data with its own (unless custom AMI, then you own it entirely) You own it entirely — always
Node health checks AWS monitors — unhealthy nodes auto-replaced Only ASG health checks (EC2-level)
Shows in EKS console? Yes — under Compute → Node Groups Nodes appear, but no "node group" entity in EKS API
Launch template Optional (AWS creates one if you don't) Required — you create and manage it
Node IAM role Specified in EKS API → auto-authorized You must authorize via access entries or aws-auth ConfigMap
Node tagging Automatic You must tag: kubernetes.io/cluster/<name> = owned
🔧 For your bootc AMI Your bootc image is its own OS — not AL2023, not Bottlerocket, not Ubuntu. This means managed node groups may not work unless your bootc image can satisfy the bootstrap contract of one of the supported AMI families (e.g. running nodeadm like AL2023). Self-managed node groups are the safe bet — they have no AMI family restriction.

Sources: AWS Docs: Managed node groups, AWS Docs: Self-managed nodes

The Bootstrap Contract

For any EC2 instance to become an EKS node, it must satisfy a contract — a set of requirements that allow kubelet on the instance to register with the EKS API server. Here's every piece:

What must be on the AMI (baked in)

ComponentPurposePath (AL2023 reference)
kubelet The node agent that registers with the API server and runs pods /usr/bin/kubelet
containerd Container runtime /usr/bin/containerd
aws-iam-authenticator Authenticates kubelet to the EKS API using the node's IAM role /usr/local/bin/aws-iam-authenticator
CNI plugins Networking binaries for pod networking /opt/cni/bin/
ECR credential provider Lets kubelet pull images from ECR /etc/eks/image-credential-provider/
iptables Required for kube-proxy networking rules /usr/sbin/iptables
A bootstrap mechanism Something that configures and starts kubelet at boot nodeadm (AL2023) or /etc/eks/bootstrap.sh (legacy AL2)

Source: GitHub: amazon-eks-ami, AWS Docs: Build a custom EKS AMI

What must be provided at boot (via user data)

The bootstrap mechanism — whether nodeadm or bootstrap.sh — needs to configure kubelet with four pieces of cluster-specific information:

ParameterWhat it isHow to retrieve it
Cluster name The name of your EKS cluster You already know this
API server endpoint The HTTPS URL of the cluster's API server aws eks describe-cluster --name <name> --query cluster.endpoint
Certificate authority Base64-encoded CA cert for TLS to the API server aws eks describe-cluster --name <name> --query cluster.certificateAuthority.data
Service CIDR The cluster's Kubernetes service IP range (used to derive cluster DNS IP) aws eks describe-cluster --name <name> --query cluster.kubernetesNetworkConfig.serviceIpv4Cidr
🔑 Key insight These four parameters are the entire cluster-specific contract. Everything else is either baked into the AMI or provided by AWS infrastructure (IAM role via instance profile, VPC networking via ENI). If your bootc AMI can accept these four values and use them to start kubelet correctly, it will join the cluster.

What must exist in AWS (infrastructure)

  • Node IAM role with policies: AmazonEKSWorkerNodePolicy, AmazonEKS_CNI_Policy, AmazonEC2ContainerRegistryReadOnly
  • Instance profile attached to the EC2 instance, referencing the node IAM role
  • Security group allowing traffic between the node and the control plane (the cluster security group)
  • Node authorization — the node's IAM role must be granted Kubernetes permissions via access entries or aws-auth ConfigMap
  • EC2 tag: kubernetes.io/cluster/<cluster-name> = owned

nodeadm: The Modern Bootstrap Tool

Since K8s 1.33+, only AL2023 and Bottlerocket AMIs are published by AWS. AL2023 uses nodeadm, which uses a YAML-based NodeConfig object. This is now the standard way to bootstrap EKS nodes.

Minimal NodeConfig

# This is the minimum viable NodeConfig
---
apiVersion: node.eks.aws/v1alpha1
kind: NodeConfig
spec:
  cluster:
    name: my-cluster
    apiServerEndpoint: https://ABCDEF1234567890.gr7.us-east-2.eks.amazonaws.com
    certificateAuthority: LS0tLS1CRUdJTi...base64...==
    cidr: 10.100.0.0/16

NodeConfig with kubelet customization

---
apiVersion: node.eks.aws/v1alpha1
kind: NodeConfig
spec:
  cluster:
    name: my-cluster
    apiServerEndpoint: https://ABCDEF1234567890.gr7.us-east-2.eks.amazonaws.com
    certificateAuthority: LS0tLS1CRUdJTi...base64...==
    cidr: 10.100.0.0/16
  kubelet:
    config:
      maxPods: 110
      shutdownGracePeriod: 30s
    flags:
      - "--node-labels=role=worker,env=dev"

Full NodeConfig API

The NodeConfigSpec has these top-level sections:

SectionPurpose
clusterCluster connection info (name, endpoint, CA, CIDR) — required
kubeletKubelet configuration (config for KubeletConfiguration fields, flags for CLI flags)
containerdInline containerd TOML config, merged with defaults
instanceInstance-level options (local storage strategy)
featureGatesEnable/disable application features

Source: nodeadm API reference, GitHub: amazon-eks-ami/nodeadm

How nodeadm runs on AL2023

nodeadm runs as two systemd services, in this order:

Boot sequence on an AL2023 EKS node: 1. EC2 instance starts 2. cloud-init begins 3. ┌─ nodeadm-config.service ──────────────────────┐ │ Runs BEFORE user data │ │ Sets up baseline containerd + kubelet config │ └───────────────────────────────────────────────┘ 4. ┌─ cloud-init user data ────────────────────────┐ │ Your shell scripts run here │ │ You can write files to /etc/eks/nodeadm.d/ │ └───────────────────────────────────────────────┘ 5. ┌─ nodeadm-run.service ─────────────────────────┐ │ Runs AFTER user data │ │ Merges all NodeConfig sources │ │ Starts containerd, kubelet, etc. │ │ Node registers with API server │ └───────────────────────────────────────────────┘ 6. Node becomes Ready
⚠️ Do NOT run nodeadm init manually On EKS-optimized AL2023 AMIs, nodeadm init is already wired into the systemd boot sequence. Running it again in user data will break things (misconfigured ENIs, duplicate bootstrap). If you need to add config, write YAML files to /etc/eks/nodeadm.d/ — they'll be merged automatically.

Source: AWS Docs: AL2023 nodeadm

User Data Formats

How you pass the NodeConfig to the instance depends on your setup:

Option A: Plain NodeConfig (self-managed, simple case)

If your AMI expects nodeadm to read from user data and you have no shell scripts to run:

---
apiVersion: node.eks.aws/v1alpha1
kind: NodeConfig
spec:
  cluster:
    name: my-cluster
    apiServerEndpoint: https://...
    certificateAuthority: LS0t...
    cidr: 10.100.0.0/16

Option B: MIME multi-part (when mixing NodeConfig + shell scripts)

If you need to run shell scripts and provide NodeConfig, use MIME format:

MIME-Version: 1.0
Content-Type: multipart/mixed; boundary="BOUNDARY"

--BOUNDARY
Content-Type: application/node.eks.aws

---
apiVersion: node.eks.aws/v1alpha1
kind: NodeConfig
spec:
  cluster:
    name: my-cluster
    apiServerEndpoint: https://...
    certificateAuthority: LS0t...
    cidr: 10.100.0.0/16
  kubelet:
    flags:
      - "--node-labels=role=worker"

--BOUNDARY
Content-Type: text/x-shellscript; charset="us-ascii"

#!/bin/bash
echo "Custom setup running before nodeadm-run"
# install packages, configure things, etc.

--BOUNDARY--
ℹ️ MIME merging You can include multiple application/node.eks.aws MIME parts, each with a partial NodeConfig. nodeadm merges them in order — later values win. This lets you separate cluster config from kubelet customization.

Source: nodeadm examples, AWS re:Post: Custom data with AL2023

Launch Templates: The Glue

A launch template is an EC2 resource that ties together the AMI, instance type, user data, and networking config. Every node group (managed or self-managed) uses one.

# Key fields in a launch template (conceptual)
LaunchTemplate:
  ImageId: ami-0123456789abcdef0    # Your AMI
  InstanceType: t3.medium
  UserData: <base64-encoded>        # NodeConfig or MIME doc
  SecurityGroupIds:
    - sg-cluster-security-group     # Must include cluster SG
  IamInstanceProfile:
    Arn: arn:aws:iam::...:instance-profile/node-role
  TagSpecifications:
    - Tags:
        - Key: kubernetes.io/cluster/my-cluster
          Value: owned

When using eksctl, you rarely create launch templates manually — eksctl generates them. But understanding the structure matters because you'll need to customize the AMI ID and user data for your bootc image.

eksctl Config: Self-Managed with Custom AMI

Here's what an eksctl config looks like when using a custom AMI with a self-managed node group:

apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig

metadata:
  name: my-cluster
  region: us-east-2
  version: "1.36"

# Self-managed node group with custom AMI
nodeGroups:
  - name: custom-nodes
    instanceType: t3.medium
    desiredCapacity: 2
    minSize: 1
    maxSize: 4
    amiFamily: AmazonLinux2023      # tells eksctl the bootstrap style
    ami: ami-0123456789abcdef0      # your custom AMI ID
    ssh:
      allow: false

For AL2023-family AMIs, eksctl automatically generates the minimal NodeConfig user data with the cluster's endpoint, CA, and CIDR. You don't need overrideBootstrapCommand — eksctl injects the config into the launch template for you.

If you need to customize the NodeConfig, use overrideBootstrapCommand:

nodeGroups:
  - name: custom-nodes
    instanceType: t3.medium
    desiredCapacity: 2
    amiFamily: AmazonLinux2023
    ami: ami-0123456789abcdef0
    overrideBootstrapCommand: |
      apiVersion: node.eks.aws/v1alpha1
      kind: NodeConfig
      spec:
        instance:
          localStorage:
            strategy: RAID0
        kubelet:
          config:
            maxPods: 110

This custom config is prepended to the user data and merged by nodeadm with the default config that eksctl generates.

Source: eksctl: Custom AMI support, eksctl: Node bootstrapping

What This Means for Your Bootc AMI

🔧 Connecting the dots

Your bootc container image → AMI must satisfy this contract:

  1. Bake in the binaries: kubelet, containerd, aws-iam-authenticator, CNI plugins, ECR credential provider, iptables
  2. Bake in a bootstrap mechanism: Either include nodeadm and wire it into systemd (like AL2023), or build your own mechanism that reads the 4 cluster parameters and configures + starts kubelet
  3. Accept cluster params at boot: Via user data (EC2 IMDS) containing a NodeConfig YAML, or via another mechanism
  4. Deploy as self-managed node group: Use eksctl's nodeGroups (not managedNodeGroups) with ami: <your-ami-id>. Set amiFamily: AmazonLinux2023 if your bootc image uses nodeadm

The amazon-eks-ami repo is the reference implementation. When building your bootc image, study what it installs and how it wires up nodeadm. The most relevant parts are:

  • Packer templates — what packages and binaries get installed
  • Runtime directory — the systemd units for nodeadm-config and nodeadm-run
  • nodeadm source — the Go binary that processes NodeConfig

Check Your Understanding

Question 1: Your bootc AMI doesn't fit any of the managed node group AMI families. What's your best option?

✅ Correct! Self-managed node groups accept any AMI — there's no AMI family restriction. You control the launch template, user data, and bootstrap process entirely.
❌ Not quite. Self-managed node groups are the answer — they have no AMI family restriction. Managed node groups require one of the recognized families. There's no "Custom" family option.

Question 2: What are the four cluster-specific parameters that must be provided to the bootstrap mechanism?

✅ Correct! The four parameters are: cluster name, API server endpoint, certificate authority (base64-encoded CA), and service CIDR. These are exactly what goes into the NodeConfig.spec.cluster section.
❌ Not quite. The four parameters are: cluster name, API server endpoint, certificate authority (base64 CA data), and service CIDR. VPC/subnet/SG are handled at the infrastructure level, not the bootstrap level.

Question 3: Why does AL2023's nodeadm require you to explicitly pass the API endpoint and CA, when the old bootstrap.sh auto-discovered them?

✅ Correct! The old bootstrap.sh called DescribeCluster at boot to fetch these values. At scale (hundreds of nodes launching simultaneously), this caused API throttling. nodeadm avoids this by requiring the values upfront in user data.
❌ Not quite. The reason is API throttling — when hundreds of nodes launch simultaneously, each one calling DescribeCluster would overwhelm the API. Providing the values statically in user data avoids this.
