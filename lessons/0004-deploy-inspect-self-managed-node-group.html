Lesson 4: Deploy & Inspect a Self-Managed Node Group

~35 min · Requires AWS account · Hands-on with real infrastructure · Remember to clean up!

You've learned the theory — node group types, the bootstrap contract, nodeadm, NodeConfig. Now you'll deploy a self-managed node group with an official EKS AMI, then inspect every layer of what eksctl created. By the end, you'll know exactly what to change when you swap in your bootc AMI.

🎯 What you'll be able to do after this lesson
  • Deploy a self-managed node group using eksctl
  • Inspect the launch template and decode the user data (NodeConfig)
  • Verify node registration and inspect kubelet's configuration
  • Shell into the node and inspect nodeadm logs, kubelet config, containerd, and the bootstrap artifacts
  • Trace the full path from eksctl config → CloudFormation → launch template → running node

The Config File

Create a file called cluster-selfmanaged.yaml. This is your complete cluster config — control plane plus one self-managed node group:

1 Create the eksctl config file

# cluster-selfmanaged.yaml
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig

metadata:
  name: jlebon
  region: ca-central-1
  version: "1.36"

# No managedNodeGroups — we're using nodeGroups (self-managed)
nodeGroups:
  - name: jlebon-selfmanaged
    instanceType: t3.medium
    desiredCapacity: 1       # one node is enough to learn
    minSize: 1
    maxSize: 2
    amiFamily: AmazonLinux2023
    privateNetworking: true  # nodes in private subnets
    ssh:
      enableSsm: true       # access the node via SSM Session Manager

Key differences from lesson 2's config:

  • nodeGroups instead of managedNodeGroups — this creates a self-managed node group
  • amiFamily: AmazonLinux2023 — tells eksctl to use nodeadm-style bootstrap
  • desiredCapacity: 1 — keep costs minimal while learning
  • ssh.enableSsm: true — enables SSM Session Manager access so we can shell into the node (works even in private subnets — no SSH key or bastion needed)

Source: eksctl: Nodegroups, eksctl: Node bootstrapping

Deploy the Cluster

2 Create the cluster

eksctl create cluster -f cluster-selfmanaged.yaml

This will take ~15 minutes. While it runs, eksctl creates two CloudFormation stacks:

StackWhat it creates
eksctl-jlebon-cluster VPC, subnets, NAT gateway, internet gateway, EKS control plane, cluster security group, IAM roles
eksctl-jlebon-nodegroup-jlebon-selfmanaged Launch template, Auto Scaling Group, node IAM role, instance profile, node security group
ℹ️ Self-managed vs managed: what's different here With managed node groups (managedNodeGroups), eksctl tells the EKS API to create the node group — AWS creates the ASG and launch template. With self-managed (nodeGroups), eksctl creates all of this itself via CloudFormation. The EKS API never knows about a "node group" entity — it just sees individual EC2 instances joining the cluster.

Watch for the final output line:

EKS cluster "jlebon" in "ca-central-1" region is ready

Verify the Node Joined

3 Check node status

# List nodes — you should see one in Ready state
kubectl get nodes -o wide

# Expected output (something like):
# NAME                                       STATUS   ROLES    AGE   VERSION   INTERNAL-IP   ...
# ip-192-168-xx-xx.ca-central-1.compute.internal Ready    <none>   2m    v1.36.x   192.168.x.x   ...

Note several things about the node:

  • ROLES is <none> — self-managed nodes don't get a role label by default (managed nodes get node)
  • NAME is the private DNS name — because the node is in a private subnet
  • VERSION matches your cluster version — eksctl selected the right AMI

4 Inspect node labels

kubectl get node -o json | jq '.items[0].metadata.labels'

Look for the eksctl-injected labels:

{
  "alpha.eksctl.io/cluster-name": "jlebon",
  "alpha.eksctl.io/nodegroup-name": "jlebon-selfmanaged",
  "eks.amazonaws.com/nodegroup": "jlebon-selfmanaged",
  "eks.amazonaws.com/capacityType": "ON_DEMAND",
  // ... plus standard kubernetes.io labels
}

These labels were injected via --node-labels in the kubelet flags — which eksctl set in the NodeConfig. You'll see exactly where in the next step.

Inspect the Launch Template

This is the most important part: understanding exactly what eksctl built.

5 Find and inspect the launch template

# Find the launch template — eksctl names it eksctl-<cluster>-nodegroup-<ng-name>
aws ec2 describe-launch-templates \
  --filters "Name=launch-template-name,Values=eksctl-jlebon-nodegroup-*" \
  --query 'LaunchTemplates[].{Name:LaunchTemplateName,ID:LaunchTemplateId}' \
  --output table

# Alternative: get it from the CloudFormation stack (always works)
aws cloudformation describe-stack-resources \
  --stack-name eksctl-jlebon-nodegroup-jlebon-selfmanaged \
  --query 'StackResources[?ResourceType==`AWS::EC2::LaunchTemplate`].PhysicalResourceId' \
  --output text

Now get the full launch template data:

# Replace LT_ID with the ID from above
LT_ID="lt-0xxxxxxxxxxxxxxxxx"

aws ec2 describe-launch-template-versions \
  --launch-template-id "$LT_ID" \
  --versions '$Default' \
  --query 'LaunchTemplateVersions[0].LaunchTemplateData' \
  --output json | jq '.'

You'll see the full launch template — AMI ID, instance type, security groups, IAM instance profile, and the base64-encoded user data.

6 Decode the user data — see the NodeConfig

This is the moment of truth. The user data contains the NodeConfig that nodeadm uses to bootstrap the node:

# Decode the user data from the launch template
aws ec2 describe-launch-template-versions \
  --launch-template-id "$LT_ID" \
  --versions '$Default' \
  --query 'LaunchTemplateVersions[0].LaunchTemplateData.UserData' \
  --output text | base64 -d

You should see something like:

MIME-Version: 1.0
Content-Type: multipart/mixed; boundary=//

--//
Content-Type: application/node.eks.aws

apiVersion: node.eks.aws/v1alpha1
kind: NodeConfig
spec:
  cluster:
    apiServerEndpoint: https://ABCDEF1234.sk1.ca-central-1.eks.amazonaws.com
    certificateAuthority: LS0tLS1CRUdJTi...long-base64...
    cidr: 10.100.0.0/16
    name: jlebon
  kubelet:
    config:
      clusterDNS:
      - 10.100.0.10
    flags:
    - "--node-labels=alpha.eksctl.io/cluster-name=jlebon,..."

--//--
🔑 This is the bootstrap contract in action

Everything you learned in Lesson 3 is right here. The four cluster parameters (apiServerEndpoint, certificateAuthority, cidr, name) are injected by eksctl. The kubelet config sets clusterDNS (derived from the CIDR — .10 of the service range). The kubelet flags inject the node labels. This is all that nodeadm needs to join the node to the cluster.

Inspect the IAM Setup

7 Find the node IAM role and its policies

# List IAM roles created by eksctl for this cluster
aws iam list-roles \
  --query "Roles[?contains(RoleName, 'jlebon-nodegroup')].[RoleName,Arn]" \
  --output table

# Get the attached policies (replace ROLE_NAME)
ROLE_NAME="eksctl-jlebon-nodegroup-ng-se-NodeInstanceRole-XXXX"
aws iam list-attached-role-policies --role-name "$ROLE_NAME" --output table

You should see the three required policies from Lesson 3:

PolicyWhy it's needed
AmazonEKSWorkerNodePolicy Allows kubelet to call EKS APIs (describe cluster, etc.)
AmazonEKS_CNI_Policy Allows the VPC CNI plugin to manage ENIs for pod networking
AmazonEC2ContainerRegistryReadOnly Allows pulling container images from ECR
AmazonSSMManagedInstanceCore Allows SSM Session Manager access (eksctl adds this by default)

8 Check how the node is authorized in Kubernetes

For self-managed nodes, eksctl must explicitly authorize the node IAM role. Check the access entries:

# List access entries for the cluster
aws eks list-access-entries --cluster-name jlebon --output json | jq '.'

# Describe the node role's access entry (replace with actual ARN)
NODE_ROLE_ARN="arn:aws:iam::ACCOUNT:role/eksctl-jlebon-nodegroup-..."
aws eks describe-access-entry \
  --cluster-name jlebon \
  --principal-arn "$NODE_ROLE_ARN" \
  --output json | jq '.'

The access entry's type should be EC2_LINUX — this tells EKS to authorize EC2 instances with this IAM role as Kubernetes nodes.

ℹ️ Access entries vs aws-auth ConfigMap Older clusters used the aws-auth ConfigMap to map IAM roles to Kubernetes identities. Access entries are the newer mechanism — they're managed via the EKS API rather than a ConfigMap. eksctl uses access entries by default on new clusters. For your bootc AMI work, this means the IAM role attached to your custom nodes must have an access entry — eksctl handles this, but if you're launching EC2 instances manually, you'll need to create it yourself.

Source: AWS Docs: EKS access entries

Inspect the Auto Scaling Group

9 Find the ASG and its configuration

# Find the ASG
aws autoscaling describe-auto-scaling-groups \
  --query "AutoScalingGroups[?contains(AutoScalingGroupName, 'jlebon')].\
{Name:AutoScalingGroupName,Desired:DesiredCapacity,Min:MinSize,Max:MaxSize,\
LT:MixedInstancesPolicy.LaunchTemplate.LaunchTemplateSpecification.LaunchTemplateId}" \
  --output table

Note that the ASG references the same launch template you inspected earlier. This is the chain: eksctl config → CloudFormation → launch template + ASG → EC2 instance → nodeadm reads user data → kubelet registers with EKS.

Inspect the CloudFormation Stack

10 See every resource eksctl created for the node group

aws cloudformation describe-stack-resources \
  --stack-name eksctl-jlebon-nodegroup-jlebon-selfmanaged \
  --query 'StackResources[].{Type:ResourceType,LogicalID:LogicalResourceId,Status:ResourceStatus}' \
  --output table

You'll see all the resources that make up a self-managed node group:

Resource TypeWhat it is
AWS::EC2::LaunchTemplateThe launch template (AMI + user data + instance config)
AWS::AutoScaling::AutoScalingGroupThe ASG that manages EC2 instance lifecycle
AWS::IAM::RoleThe node IAM role
AWS::IAM::InstanceProfileThe instance profile wrapping the IAM role
AWS::EC2::SecurityGroupNode security group (traffic between nodes)
AWS::EC2::SecurityGroupIngress/EgressRules allowing node ↔ control plane communication
AWS::EKS::AccessEntryAuthorizes the node IAM role in Kubernetes
🔧 For your bootc AMI

When you deploy your bootc AMI, the only thing that changes in this stack is the AMI ID in the launch template and possibly the user data. Everything else — IAM, security groups, ASG, access entry — stays the same. This is why understanding the infrastructure layer separately from the bootstrap layer is critical.

Run a Workload

11 Deploy a simple pod to verify the node is fully functional

# Create a simple nginx deployment
kubectl create deployment nginx --image=nginx:latest --replicas=1

# Wait for it to be ready
kubectl rollout status deployment/nginx

# Verify it's running on your self-managed node
kubectl get pods -o wide

# Check that the pod can reach the internet (DNS works, networking works)
kubectl exec deploy/nginx -- curl -s -o /dev/null -w "%{http_code}" https://aws.amazon.com

If you get a 200 back, your self-managed node is fully functional — kubelet is running pods, the CNI plugin assigned a pod IP, kube-proxy set up iptables rules, and DNS resolution works.

Inspect the Node from Inside

You've seen the node from the outside — launch template, IAM, CloudFormation. Now shell into the node and see what a working EKS node looks like from the inside. This is the most directly useful part for your bootc AMI work: it's a checklist of everything your image needs to replicate.

12 Connect to the node via SSM Session Manager

# Get the instance ID of your node
INSTANCE_ID=$(aws ec2 describe-instances \
  --filters "Name=tag:alpha.eksctl.io/cluster-name,Values=jlebon" \
            "Name=instance-state-name,Values=running" \
  --query 'Reservations[0].Instances[0].InstanceId' \
  --output text)

echo "$INSTANCE_ID"

# Start an SSM session (opens an interactive shell on the node)
aws ssm start-session --target "$INSTANCE_ID"

You'll get a shell as ssm-user. Switch to root to inspect system services:

sudo -i
ℹ️ Why SSM instead of SSH? SSM Session Manager works through the SSM agent (already installed on AL2023), routes through AWS APIs, and doesn't require a bastion host, SSH key, or port 22 in the security group. It works even for nodes in private subnets with no public IP — exactly our setup. The AmazonSSMManagedInstanceCore policy you saw earlier is what makes this possible.

Source: AWS Docs: SSM Session Manager

13 Read the nodeadm boot logs

nodeadm runs as two systemd services in sequence. These logs tell you exactly what happened during bootstrap:

# Phase 1: nodeadm-config (runs BEFORE user data)
# Sets up baseline containerd + kubelet config
journalctl -u nodeadm-config --no-pager

# Phase 2: nodeadm-run (runs AFTER user data)
# Starts containerd, kubelet, and registers the node
journalctl -u nodeadm-run --no-pager

In the nodeadm-config output, look for:

  • Loading configuration.. {"configSource": ["imds://user-data"]} — it read the NodeConfig from EC2 user data via IMDS
  • Fetched kubelet version — the kubelet binary version it found on the AMI

In the nodeadm-run output, look for:

  • Containerd starting
  • Kubelet configuration being written
  • Sandbox image being pulled from ECR
  • Kubelet starting and registering
# Also check cloud-init logs to see the full boot sequence
journalctl -u cloud-init --no-pager | head -50

# And the cloud-init output log (captures stdout of user-data scripts)
cat /var/log/cloud-init-output.log 2>/dev/null | tail -30
🔧 For your bootc AMI

Your bootc image won't have nodeadm (or you may choose not to use it). But this log output is a blueprint: it shows the sequence of operations a bootstrap mechanism must perform. Your custom bootstrap script needs to do the same things in the same order: read config → configure runtime → configure kubelet → start services.

14 Inspect kubelet and containerd

Kubelet

# Kubelet service status
systemctl status kubelet

# Kubelet configuration (written by nodeadm)
cat /etc/kubernetes/kubelet/config.json | jq .

# Kubelet kubeconfig (how kubelet authenticates to the API server)
cat /var/lib/kubelet/kubeconfig

# Kubelet environment variables (flags passed at startup)
cat /etc/eks/kubelet/environment

# The CA certificate kubelet uses for TLS
ls -la /etc/kubernetes/pki/ca.crt

Key things to notice in the kubelet config:

  • clusterDNS — set to the .10 address of the service CIDR (e.g. 10.100.0.10)
  • containerRuntimeEndpoint — points to containerd's Unix socket
  • authentication — uses webhook (IAM-based via aws-iam-authenticator)
  • serverTLSBootstrap — kubelet bootstraps its own serving certificate

Containerd

# Containerd service status
systemctl status containerd

# Containerd config (written by nodeadm, then containerd merges defaults)
cat /etc/containerd/config.toml

# Check the sandbox (pause) image — pulled from ECR
crictl images | grep pause
🔧 For your bootc AMI

Your image uses CRI-O instead of containerd. The kubelet config's containerRuntimeEndpoint will need to point to CRI-O's socket (unix:///var/run/crio/crio.sock) instead of containerd's. The rest of the kubelet config — clusterDNS, authentication, TLS bootstrap — stays the same regardless of runtime. This is one of the concrete things you now know to change.

15 Inspect the bootstrap artifacts

Binaries on the AMI

# Every binary from the bootstrap contract (Lesson 3) — verify they're all here
ls -la /usr/bin/kubelet
ls -la /usr/bin/containerd
ls -la /usr/local/bin/aws-iam-authenticator
ls -la /opt/cni/bin/
ls -la /etc/eks/image-credential-provider/ecr-credential-provider
ls -la /usr/sbin/iptables
ls -la /usr/bin/nodeadm

User data as the node received it

# Read the raw user data from IMDS — this is what nodeadm parsed at boot
TOKEN=$(curl -sX PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 60")
curl -s -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/user-data

This should match exactly what you decoded from the launch template in step 6. The node received the NodeConfig through EC2 Instance Metadata Service (IMDS), and nodeadm parsed it from there.

Nodeadm's cached config

# nodeadm caches its resolved config here after boot
cat /run/eks/nodeadm/config.json 2>/dev/null | jq . || echo "(not cached on this version)"

Systemd units

# The two nodeadm systemd services that orchestrate the bootstrap
systemctl cat nodeadm-config
echo "---"
systemctl cat nodeadm-run

# See the ordering: nodeadm-config runs before cloud-init,
# nodeadm-run runs after cloud-init
systemctl show -p After nodeadm-run.service | tr ' ' '\n' | head -10

Kernel modules and sysctl

# Kernel modules loaded for containerd networking
cat /etc/modules-load.d/containerd.conf

# Sysctl settings for pod networking (IP forwarding, etc.)
cat /etc/sysctl.d/99-nodeadm.conf
🔑 What you're looking at is a complete inventory

Every file and binary you've just inspected is something your bootc AMI either needs to include or replace with an equivalent. The binaries (kubelet, CNI, iam-authenticator) are universal. The configs (kubelet config, runtime config, sysctl) need to be generated by your bootstrap mechanism. The systemd units are the orchestration glue. This is the reference implementation for "what does a working EKS node look like?"

# When you're done exploring, exit the SSM session
exit  # exit root shell
exit  # exit SSM session

Sources: nodeadm overview, AWS Docs: nodeadm reference (artifact paths), AWS Docs: AL2023 nodeadm

The Full Chain — What You Just Deployed

eksctl config (YAML) │ ▼ CloudFormation stacks │ ├── Cluster stack │ ├── VPC + subnets + NAT/IGW │ ├── EKS control plane │ └── Cluster security group │ └── Node group stack ├── IAM role + instance profile ├── Node security group + ingress/egress rules ├── Launch template │ ├── AMI ID ◄── THIS is what you change for bootc │ ├── Instance type │ ├── Security groups │ ├── Instance profile │ └── User data (NodeConfig) ◄── AND THIS ├── Auto Scaling Group → EC2 instances └── Access entry (node IAM role → Kubernetes auth) At boot: EC2 instance starts → cloud-init → nodeadm reads NodeConfig → configures + starts containerd → configures + starts kubelet → kubelet authenticates via IAM → registers with EKS API server → node becomes Ready

Console Walkthrough

16 See the self-managed node in the EKS console

  1. Open the EKS Console
  2. Click jlebon → Compute tab
  3. Notice: under Node groups, there's nothing — self-managed nodes don't appear as an EKS "node group" entity
  4. Click Nodes — your node appears here because kubelet registered it, but EKS doesn't know it's part of a "group"
  5. Click the node name → you can see its instance ID, labels, conditions, and running pods

Now compare in the EC2 Console:

  1. Find your instance (filter by tag alpha.eksctl.io/cluster-name = jlebon)
  2. Click Actions → Instance settings → View/change user data — you'll see the same NodeConfig MIME document you decoded earlier
  3. Click the Security tab → note the IAM role and security groups
  4. Click the Networking tab → note the private IP and the VPC subnet

Mapping to Your Bootc AMI

Now that you've seen every piece, here's the concrete translation for your bootc work:

What eksctl did with the official AMIWhat you'll need for bootc
Used amiFamily: AmazonLinux2023 to pick the right AMI Use ami: ami-YOUR-BOOTC-ID to specify your AMI explicitly
Generated NodeConfig user data for nodeadm Either include nodeadm in your bootc image, or write a custom bootstrap that reads the same 4 parameters from IMDS user data
nodeadm started containerd + kubelet Your bootc image uses CRI-O — you'll need a bootstrap script that configures and starts CRI-O + kubelet (bypassing nodeadm's containerd assumptions)
eksctl set amiFamily to control user data format With a truly custom bootstrap, you may need overrideBootstrapCommand or a raw user data approach
Everything else (IAM, SG, ASG, access entry) — unchanged Same — no changes needed
🔧 The key realization

The infrastructure layer (IAM, networking, ASG, access entries) is AMI-agnostic. It doesn't care what OS is running. The only AMI-specific parts are the launch template AMI ID and the user data format. Your bootc AMI work is essentially: build the right AMI, write the right user data handler, and plug them into the same infrastructure.

Clean Up

17 Delete everything

# Delete the workload first (optional — eksctl will clean up anyway)
kubectl delete deployment nginx

# Delete the cluster and all associated resources
eksctl delete cluster -f cluster-selfmanaged.yaml --wait

# Verify nothing is left (NAT gateways cost money!)
aws cloudformation list-stacks \
  --query "StackSummaries[?contains(StackName,'jlebon') && StackStatus!='DELETE_COMPLETE']" \
  --output table
⚠️ Cost reminder The biggest cost driver here is the NAT gateway (~$0.045/hr) and the EC2 instance. The EKS control plane is $0.10/hr. If you leave this running for a day, it's roughly $4-5. Always clean up when done.
