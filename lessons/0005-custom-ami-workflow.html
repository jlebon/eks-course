Finding AMI IDs via SSM Parameter Store

When eksctl auto-selects an AMI (as in Lesson 4), it queries SSM Parameter Store behind the scenes. You need to know how to do this yourself — both to pin specific AMI versions and to understand what AMI your cluster is actually running.

1 Query the recommended AMI for your cluster version

# Pattern:
# /aws/service/eks/optimized-ami/<k8s-version>/<ami-type>/recommended/image_id

# Get the latest recommended AL2023 x86_64 AMI for Kubernetes 1.36
aws ssm get-parameter \
  --name /aws/service/eks/optimized-ami/1.36/amazon-linux-2023/x86_64/standard/recommended/image_id \
  --region ca-central-1 \
  --query "Parameter.Value" \
  --output text

This returns a single AMI ID like ami-0abcdef1234567890. Save it — you'll use it in the cluster config.

# You can also get the full metadata (AMI name, release version, etc.)
aws ssm get-parameter \
  --name /aws/service/eks/optimized-ami/1.36/amazon-linux-2023/x86_64/standard/recommended \
  --region ca-central-1 \
  --query "Parameter.Value" \
  --output text | jq .

This returns:

{
  "schema_version": "2",
  "image_id": "ami-0abcdef1234567890",
  "image_name": "amazon-eks-node-al2023-x86_64-standard-1.36-v20260610",
  "release_version": "1.36.0-20260610"
}

The release_version encodes the Kubernetes minor version and the AMI build date. When you need to pin a specific version (e.g. for reproducible deploys), you can look up older versions too.

The ami: Field

In Lesson 4's config, there was no ami: field — eksctl auto-selected the AMI based on amiFamily, region, and Kubernetes version. Adding ami: tells eksctl to use your specific AMI instead.

2 Create a config with an explicit AMI

Create cluster-custom-ami.yaml:

# cluster-custom-ami.yaml
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig

metadata:
  name: jlebon
  region: ca-central-1
  version: "1.36"

nodeGroups:
  - name: custom-ami-ng
    instanceType: t3.medium
    desiredCapacity: 1
    minSize: 1
    maxSize: 2
    amiFamily: AmazonLinux2023
    ami: ami-REPLACE_WITH_SSM_RESULT  # ← from step 1
    privateNetworking: true
    ssh:
      enableSsm: true

The only difference from Lesson 4's config is the ami: line. Everything else — IAM, NodeConfig generation, launch template, ASG — works exactly the same.

🔑 Why this matters

You're using the official AMI here, but specifying it explicitly. This exercises the exact same code path your bootc AMI will use. When your bootc AMI is ready, you just swap ami-REPLACE_WITH_SSM_RESULT for ami-YOUR_BOOTC_AMI. Nothing else changes.

Version Matching

The Kubernetes version baked into the AMI must be compatible with the cluster's version. This is a common source of failures with custom AMIs.

ScenarioResult
AMI kubelet version == cluster version ✅ Works — ideal case
AMI kubelet is 1 minor version behind cluster ✅ Works — Kubernetes supports n-1 kubelet (skew policy)
AMI kubelet is 2+ minor versions behind cluster ❌ Fails — kubelet won't register
AMI kubelet is newer than cluster ❌ Fails — not supported by skew policy
🔧 For your bootc AMI

The kubelet version in your bootc image must match or be one minor version behind the EKS cluster. When you upgrade your EKS cluster to a new Kubernetes version, you'll need to rebuild your bootc image with the matching kubelet. The SSM query in step 1 is how you check which kubelet version the official AMI ships — your bootc image should match it.

Source: Kubernetes: Version skew policy

Deploy and Verify

3 Create the cluster

eksctl create cluster -f cluster-custom-ami.yaml

Watch the eksctl output. You should see it reference your specific AMI ID in the log, rather than the phrase "resolving AMI" that you'd see with auto-selection.

4 Verify and compare with Lesson 4

# Node should be Ready
kubectl get nodes -o wide

# Confirm the AMI ID matches what you specified
INSTANCE_ID=$(aws ec2 describe-instances \
  --filters "Name=tag:alpha.eksctl.io/cluster-name,Values=jlebon" \
            "Name=instance-state-name,Values=running" \

aws ec2 describe-instances --instance-ids "$INSTANCE_ID" \
  --query 'Reservations[0].Instances[0].ImageId' --output text

This should print the exact AMI ID you specified in the config. In Lesson 4, eksctl picked this same AMI automatically — now you've pinned it.

5 Decode the user data — confirm it's identical

# Get the launch template ID
LT_ID=$(aws ec2 describe-launch-templates \
  --filters "Name=launch-template-name,Values=eksctl-jlebon-nodegroup-*" \
  --query 'LaunchTemplates[0].LaunchTemplateId' \
  --output text)

# Decode user data
aws ec2 describe-launch-template-versions \
  --launch-template-id "$LT_ID" \
  --versions '$Default' \
  --query 'LaunchTemplateVersions[0].LaunchTemplateData.UserData' \
  --output text | base64 -d

The NodeConfig should look identical to Lesson 4 — same structure, same cluster parameters, same kubelet flags. This confirms the key point: specifying ami: changes the AMI in the launch template but doesn't change how eksctl generates the user data.

Customizing the Bootstrap with overrideBootstrapCommand

For AL2023 node groups, overrideBootstrapCommand takes a NodeConfig YAML (not a shell script). This custom config is prepended to the user data and merged by nodeadm with the default config that eksctl generates. Later values win.

ℹ️ Why it's called "overrideBootstrapCommand" The name is a legacy from AL2, where this field held a shell script that replaced /etc/eks/bootstrap.sh. For AL2023, eksctl repurposed the field to hold a NodeConfig YAML that gets merged — not a script. The name is misleading but the behavior is well-defined.

6 Add custom NodeConfig

Update the node group in your config to include overrideBootstrapCommand:

nodeGroups:
  - name: custom-ami-ng
    instanceType: t3.medium
    desiredCapacity: 1
    minSize: 1
    maxSize: 2
    amiFamily: AmazonLinux2023
    ami: ami-REPLACE_WITH_SSM_RESULT
    privateNetworking: true
    ssh:
      enableSsm: true
    overrideBootstrapCommand: |
      apiVersion: node.eks.aws/v1alpha1
      kind: NodeConfig
      spec:
        kubelet:
          config:
            maxPods: 50
            shutdownGracePeriod: 30s
          flags:
            - "--node-labels=node.eks.amazonaws.com/image-type=custom"

This custom NodeConfig adds:

  • maxPods: 50 — caps the number of pods on this node
  • shutdownGracePeriod: 30s — graceful shutdown timeout
  • A custom node label node.eks.amazonaws.com/image-type=custom

To apply this to your running cluster, delete the existing node group and recreate it:

# Delete the existing node group
eksctl delete nodegroup --cluster jlebon --name custom-ami-ng --wait

# Create it again with the updated config
eksctl create nodegroup -f cluster-custom-ami.yaml

7 Verify the merged config

# Decode the new user data — you should see TWO NodeConfig blocks
LT_ID=$(aws ec2 describe-launch-templates \
  --filters "Name=launch-template-name,Values=eksctl-jlebon-nodegroup-*" \
  --query 'LaunchTemplates[0].LaunchTemplateId' \
  --output text)

aws ec2 describe-launch-template-versions \
  --launch-template-id "$LT_ID" \
  --versions '$Default' \
  --query 'LaunchTemplateVersions[0].LaunchTemplateData.UserData' \
  --output text | base64 -d

You should see a MIME document with two application/node.eks.aws parts:

  1. Your custom NodeConfig (from overrideBootstrapCommand) — listed first
  2. eksctl's default NodeConfig (cluster params, DNS, labels) — listed second

At boot, nodeadm parses both and merges them. The second (eksctl's default) wins for any conflicting fields, but your additions (maxPods, shutdownGracePeriod, extra labels) survive because they don't conflict.

⚠️ Merge order matters

nodeadm merges multiple NodeConfig blocks in order — later values win. eksctl puts your custom config first and its generated config second. This means eksctl's cluster parameters always take precedence, which is correct — you don't want to accidentally override the API server endpoint. But it also means your kubelet.flags labels get replaced by eksctl's, not appended. Use kubelet.config for settings you want to keep (it merges at the field level), and check the merged result if you're adding kubelet.flags.

8 Confirm the customization took effect on the node

kubectl get nodes --show-labels | grep image-type

# Check maxPods via the Node object
kubectl get node -o jsonpath='{.items[0].status.capacity.pods}'
echo

# Or SSM into the node and check the kubelet config directly
INSTANCE_ID=$(aws ec2 describe-instances \

The Key Insight: amiFamily = Bootstrap Protocol

This is the most important concept in this lesson:

What amiFamily DOES NOT mean: "The AMI is built on Amazon Linux 2023" What amiFamily DOES mean: "The AMI speaks the nodeadm bootstrap protocol" ┌─────────────────────────────────────────────────────────┐ │ amiFamily: AmazonLinux2023 │ │ │ │ tells eksctl: │ │ 1. Generate user data as MIME with NodeConfig YAML │ │ 2. Don't require overrideBootstrapCommand │ │ 3. If overrideBootstrapCommand is set, treat it as │ │ a NodeConfig YAML to merge (not a shell script) │ │ │ │ tells eksctl NOTHING about: │ │ - What OS is on the AMI │ │ - What packages are installed │ │ - Whether it's actually Amazon Linux │ └─────────────────────────────────────────────────────────┘

Your bootc image ships nodeadm and containerd. It understands the NodeConfig YAML format. It has the nodeadm-config and nodeadm-run systemd services wired up. Therefore, it speaks the AL2023 bootstrap protocol, and amiFamily: AmazonLinux2023 is correct — even though the underlying OS is RHEL-based.

🔧 What this means for your bootc AMI

Because you ship nodeadm + containerd:

  • Use amiFamily: AmazonLinux2023 — eksctl generates the correct user data format
  • Use ami: ami-YOUR-BOOTC-ID — eksctl uses your AMI instead of the official one
  • Use overrideBootstrapCommand if you need custom NodeConfig (labels, kubelet tuning, etc.)
  • No shell scripts, no hacks, no special handling — your bootc AMI is a drop-in replacement

The Bootc Config Template

Here's the production-ready eksctl config you'll use when your bootc AMI is ready. Save it — this is the culmination of lessons 1–5:

9 Your production config template

# cluster-bootc.yaml — Template for deploying bootc AMI nodes
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig

metadata:
  name: CLUSTER_NAME
  region: ca-central-1
  version: "1.36"           # must match kubelet in the bootc image

nodeGroups:
  - name: bootc-nodes
    instanceType: t3.medium  # adjust for your workload
    desiredCapacity: 2
    minSize: 1
    maxSize: 10
    amiFamily: AmazonLinux2023  # "speaks nodeadm" — NOT "is AL2023"
    ami: ami-YOUR_BOOTC_AMI   # your bootc image converted to AMI
    privateNetworking: true
    ssh:
      enableSsm: true        # for debugging — disable in prod if desired

    # Optional: customize the NodeConfig
    # overrideBootstrapCommand: |
    #   apiVersion: node.eks.aws/v1alpha1
    #   kind: NodeConfig
    #   spec:
    #     kubelet:
    #       config:
    #         maxPods: 110
    #       flags:
    #         - "--node-labels=image=bootc,version=1.0"

When your bootc AMI is ready, the deployment process is:

  1. Build bootc image → convert to AMI → note the AMI ID
  2. Replace ami-YOUR_BOOTC_AMI with the real AMI ID
  3. Run eksctl create cluster -f cluster-bootc.yaml
  4. Verify with kubectl get nodes
  5. If the node doesn't join: SSM in and check journalctl -u nodeadm-config and journalctl -u nodeadm-run

What Can Go Wrong with Custom AMIs

When your bootc AMI doesn't work the first time (it won't — this is normal), here's the debugging hierarchy:

SymptomLikely causeHow to check
Node never appears in kubectl get nodes kubelet never started or can't auth to API server SSM in → journalctl -u nodeadm-run, systemctl status kubelet
nodeadm-config fails Can't read user data from IMDS, or NodeConfig parse error journalctl -u nodeadm-config — look for YAML parse errors or IMDS connectivity issues
nodeadm-run fails Missing binary (containerd, kubelet, CNI), or containerd won't start journalctl -u nodeadm-run, systemctl status containerd
kubelet starts but node is NotReady CNI not configured, or missing CNI plugins in /opt/cni/bin/ kubectl describe node — check Conditions; journalctl -u kubelet
kubelet starts but gets Unauthorized Node IAM role not authorized (missing access entry) or aws-iam-authenticator missing journalctl -u kubelet — look for 401/403; check aws eks list-access-entries
kubelet rejected by API server: version skew kubelet version doesn't match cluster (>1 minor version apart) kubelet --version on the node vs kubectl version

Source: AWS Docs: Troubleshoot EKS, AWS re:Post: Nodes fail to join cluster

Check Your Understanding

Question 1: Your bootc image ships nodeadm and containerd but is built on a RHEL base. What amiFamily should you use?

✅ Correct! amiFamily means "bootstrap protocol", not "OS type". Your bootc image ships nodeadm, so it speaks the AL2023 protocol. eksctl will generate the right user data format, and nodeadm on your image will parse it.
❌ Not quite. amiFamily tells eksctl what bootstrap protocol the AMI uses — not what OS it runs. Since your bootc image ships nodeadm, it speaks the AL2023 protocol, so AmazonLinux2023 is correct. There's no RHEL or Custom family.

Question 2: Your EKS cluster is v1.36. Your bootc AMI ships kubelet v1.34. Will it work?

✅ Correct! Kubernetes version skew policy allows kubelet to be at most one minor version behind the API server. v1.34 is two behind v1.36, so it would fail. You'd need at least kubelet v1.35.
❌ The Kubernetes version skew policy allows kubelet to be at most one minor version behind the API server. v1.34 kubelet is two versions behind a v1.36 cluster, so the node would fail to register. You'd need kubelet v1.35 or v1.36.

Question 3: You set overrideBootstrapCommand with a NodeConfig that sets kubelet.flags: ["--node-labels=env=dev"]. After deploy, you don't see the env=dev label. Why?

✅ Correct! eksctl puts your config first and its generated config second. nodeadm merges them with later-wins semantics. Since eksctl's config also sets kubelet.flags (for its own labels), it replaces yours entirely. Use kubelet.config for settings that merge at the field level, or check the merged result carefully when using kubelet.flags.
❌ The issue is merge order. eksctl puts your custom config first and its generated config second. nodeadm merges with later-wins — eksctl's kubelet.flags (which sets its own labels) replaces your flags entirely. Use kubelet.config fields instead, which merge at a finer granularity.
