Lesson 1: The EKS Architecture Mental Model

~10 min · No AWS account needed · Builds foundation for all future lessons

You already know how Kubernetes works — a control plane managing worker nodes. EKS is that, but split across two distinct ownership boundaries. Understanding exactly where that boundary falls is the single most important thing for your goal of running custom bootc-based AMIs as EKS nodes.

The Split: Two VPCs, Two Owners

In a self-hosted Kubernetes cluster, you own everything. In EKS, the architecture is physically split:

🔑 Key insight The control plane runs in an AWS-managed VPC you never see. It connects to your VPC via cross-account Elastic Network Interfaces (ENIs) placed in your subnets. This is how kubelet on your nodes talks to the API server.

What AWS Manages vs. What You Manage

AWS manages You manage
API server, etcd, scheduler, controllers Worker nodes (EC2 instances)
Control plane HA (multi-AZ) Node scaling, patching, OS updates*
Control plane upgrades Data plane (node) upgrades
etcd backups VPC, subnets, security groups
API server endpoint & certs IAM roles (cluster role + node role)

*With managed node groups, AWS handles rolling updates. With self-managed nodes, you handle everything.

The Four Ways to Run Nodes

EKS offers four compute options. Understanding them matters because your bootc AMI work will likely target self-managed nodes — the option that gives you full control over the AMI.

1. EKS Auto Mode

AWS manages everything — node provisioning, scaling, patching, networking. You just deploy workloads. Least control, least ops burden. You cannot use a custom AMI.

2. Managed Node Groups

AWS manages an Auto Scaling Group of EC2 instances for you. You pick instance type and AMI family. AWS handles rolling updates and node health.

  • Custom AMIs are supported, but only from specific AMI families: AmazonLinux2023, AmazonLinux2, Bottlerocket, Ubuntu2204, Ubuntu2404
  • When you specify a custom AMI, AWS does not inject bootstrap user data — you must supply it yourself

Source: AWS Docs: Create a managed node group, eksctl: Custom AMI support

3. Self-Managed Nodes

You create and manage the Auto Scaling Group, launch template, and EC2 instances yourself. Total control over the AMI, bootstrap process, and node lifecycle.

  • This is where custom/bootc AMIs have the most freedom
  • You must ensure your AMI runs the right bootstrap process to join the cluster
  • You must tag nodes with kubernetes.io/cluster/<name> = owned

Source: AWS Docs: Self-managed nodes

4. Fargate

Serverless — each pod runs in its own micro-VM. No nodes to manage. Not relevant for custom AMI work.

⚠️ For your bootc AMI goal You'll most likely use self-managed nodes. Managed node groups might work if your bootc image can masquerade as one of the supported AMI families, but self-managed gives you the most control over bootstrapping. We'll explore this in a later lesson.

How a Node Joins the Cluster

This is the process that your bootc AMI will eventually need to replicate. Here's what happens when an EC2 instance boots as an EKS node:

  1. EC2 instance launches from an AMI, in a subnet within your VPC
  2. User data runs — either bootstrap.sh (AL2) or nodeadm (AL2023) configures kubelet with:
    • Cluster name
    • API server endpoint
    • Certificate authority (CA) data
    • Service CIDR
  3. kubelet starts and contacts the API server
  4. Node authorization — the node's IAM role must be mapped to a Kubernetes identity (via access entries or the aws-auth ConfigMap)
  5. Node registers as Ready — pods can now be scheduled on it
ℹ️ AL2 → AL2023 bootstrap shift The older /etc/eks/bootstrap.sh (shell script) is being replaced by nodeadm (a Go binary with YAML config). AL2023-based AMIs use nodeadm and require you to explicitly supply cluster metadata — it no longer auto-discovers it via API call (to avoid throttling at scale).

Source: AWS Docs: AL2023 nodeadm, GitHub: amazon-eks-ami

The Two IAM Roles

EKS requires two IAM roles. This trips up many newcomers:

Role Who uses it Key permissions
Cluster IAM Role The EKS control plane AmazonEKSClusterPolicy — lets the control plane manage AWS resources (ENIs, security groups, etc.)
Node IAM Role EC2 worker nodes AmazonEKSWorkerNodePolicy, AmazonEKS_CNI_Policy, AmazonEC2ContainerRegistryReadOnly

The node role is what allows your bootc instances to: (1) communicate with the EKS API, (2) configure pod networking via VPC CNI, and (3) pull container images from ECR.

Source: AWS Docs: Create an EKS cluster

The Three CLI Tools

You'll interact with EKS through three tools (plus the web console):

Tool What it does When to use it
eksctl High-level EKS CLI. Creates clusters via CloudFormation stacks. Quick cluster creation, node group management. Generates CloudFormation behind the scenes.
aws eks AWS CLI subcommand for EKS API. Direct API calls. Cluster creation, kubeconfig updates. More verbose but more control.
kubectl Standard K8s CLI. Everything you already know — just pointed at an EKS cluster.
🔑 eksctl vs aws eks eksctl is a convenience wrapper. Running eksctl create cluster generates CloudFormation templates that create VPC, subnets, IAM roles, security groups, and the cluster itself. aws eks create-cluster creates only the cluster — you must create the VPC, IAM roles, and everything else yourself first. Both are valid; eksctl is faster for learning, aws eks teaches you what's actually happening.

Source: AWS Docs: Getting started with EKS, eksctl.io

Check Your Understanding

Question 1: Where does the EKS control plane run?

✅ Correct! The control plane runs in an AWS-managed VPC that's completely invisible to you. It connects to your VPC through cross-account ENIs placed in your subnets.
❌ Not quite. The control plane runs in an AWS-managed VPC you never see. It connects to your VPC through cross-account ENIs. You cannot access the control plane instances directly.

Question 2: Which node type gives you full control over the AMI and bootstrap process?

✅ Correct! Self-managed nodes give you total control over the AMI, launch template, and bootstrap process. This is what you'll use for your bootc AMI.
❌ Not quite. Self-managed nodes give you total control. Managed node groups support custom AMIs but only from specific AMI families. EKS Auto Mode and Fargate don't let you bring your own AMI at all.

Question 3: What does nodeadm replace, and what info must you provide it?

✅ Correct! nodeadm replaces the legacy bootstrap.sh script. It uses a YAML NodeConfig that must include: cluster name, API server endpoint, certificate authority, and service CIDR. Unlike the old script, it does not auto-discover this via API call.
❌ Not quite. nodeadm replaces bootstrap.sh (the legacy AL2 bootstrap script). It requires a YAML NodeConfig with: cluster name, API server endpoint, certificate authority data, and service CIDR.

Question 4: What's the key difference between eksctl create cluster and aws eks create-cluster?

✅ Correct! eksctl is a convenience tool that generates CloudFormation stacks to create the VPC, subnets, IAM roles, security groups, AND the cluster. aws eks create-cluster creates only the EKS cluster itself — you must create all the supporting infrastructure separately.
❌ Not quite. eksctl is higher-level — it generates CloudFormation stacks to create VPC, IAM roles, subnets, security groups, and the cluster all at once. aws eks create-cluster creates only the cluster, so you must set up all supporting infrastructure first.

What you learned: The EKS control plane / data plane split, the four node types (and why self-managed matters for custom AMIs), how nodes bootstrap into a cluster, the two IAM roles, and the three CLI tools.
